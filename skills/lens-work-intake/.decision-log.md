# Decision Log: lens-work-intake

## 2026-05-21

- Classified as a module-based simple workflow because it creates a bounded work archive and lifecycle handoff without needing a persistent persona.
- Added `lens-work-intake` as the single recommended entry point for new Lens work while keeping existing audit, promotion, impact, topology, and reporting workflows independently invokable.
- Used file-based work memory (`memory.md`) so durable context is inspectable, editable, diffable, and reusable across related features instead of trapped in a chat thread.
- Kept intake from editing ledgers, sprint status, stories, or implementation code; governance and delivery remain owned by their existing Lens/BMad workflows.
- Chose a simple inline SKILL.md with no scripts because the workflow requires judgment and artifact writing, not deterministic parsing.
- Updated work archive frontmatter to use the shared metadata contract and `publication_state: draft` so planning state lives in files instead of branches.
