# Lens Lifecycle

## Overview

This internal skill owns the clean-room NextLens lifecycle contract for the Two-Tree Model. It reads feature records from `docs/features/<feature-id>/feature.yaml`, validates local feature-archive artifacts, and routes the next command for `full` and `express` tracks. It never delegates to or invokes `lens.core`.

## Source Model

- Feature archive: `docs/features/<feature-id>/`
- Feature record: `docs/features/<feature-id>/feature.yaml`
- Living landscape: `docs/<program>/<domain>/<service>/ledger/` or shallower service/domain ledgers
- Derived map: generated by `lens-projection-rebuild`, never edited as source truth

## Tracks

- `full`: `preplan -> businessplan -> techplan -> finalizeplan -> dev`
- `express`: `expressplan -> finalizeplan -> dev`

## Deterministic Operations

Use the stdlib script for routing and checks:

```bash
python skills/lens-lifecycle/scripts/lifecycle_ops.py --project-root {project-root} suggest-next --feature-id {feature_id}
python skills/lens-lifecycle/scripts/lifecycle_ops.py --project-root {project-root} validate-phase --feature-id {feature_id} --phase {phase}
python skills/lens-lifecycle/scripts/lifecycle_ops.py --project-root {project-root} advance-phase --feature-id {feature_id} --phase {phase}
```

## Safety Rules

- Do not read or write `lens.core` as a runtime dependency.
- Do not create planning branches or treat branch location as planning truth.
- Do not write external governance mirrors.
- Phase advancement updates only the local feature record after required local artifacts exist, unless the caller explicitly uses `--force` for recovery.
